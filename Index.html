首页必读｜报名 AI-901 的详细步骤

下面流程按 Microsoft 官方“注册并安排考试”页面整理。实际价格、可用语言、考试方式和场次会因国家/地区及考试提供商而变化，提交订单前请以页面实时显示为准。

最重要的账号建议：优先使用个人 Microsoft 帐户（MSA），不要只依赖公司/学校 Entra ID（原 Azure AD）账号。官方考试页特别提示：如果使用组织账号注册，离开组织后考试记录可能无法恢复。

准备个人 Microsoft 帐户与证件

准备可长期使用的个人 MSA、常用邮箱、手机号，以及考试当天使用的有效法定身份证件。姓名必须与证件一致；不要使用昵称、英文名替代法定姓名，除非证件上就是该姓名。

打开 AI-901 官方考试页

进入 AI-901 官方考试详情页，检查当前技能范围、语言、价格、考试政策和“安排考试”入口。官方页面当前标注英语版本已于 2026 年 4 月 15 日更新。

登录或创建 Microsoft Learn 个人资料

使用个人 MSA 登录。若 Learn 个人资料缺少法定姓名，系统会要求补充。务必逐字核对姓名与身份证件；同时检查国家/地区、时区和联系方式。

选择考试提供商

个人报考或参加一般培训：通常选择 Pearson VUE。
学生、学术机构成员：页面可用时选择 Certiport。是否显示某个提供商由考试与资格决定。

填写报考资料并提交到提供商

按页面要求填写地址、电话、联系邮箱等信息，然后选择“提交”进入 Pearson VUE 或 Certiport。官方说明这些联系资料是考试提供商所需。

选择考试方式、日期和时间

若页面提供，可选择在线监考或线下考试中心。在线考试应先运行系统预检查，并准备独立、安静、桌面整洁的考试环境；若看不到在线选项，说明当前提供商没有提供。官方说明 Certiport 目前不提供在线监考考试。

确认语言、费用和订单

确认考试代码为 AI-901、语言、时区、日期、交付方式及费用。官方考试页显示的参考价格会按国家/地区变化，因此以结算页面为准。检查优惠券/学生资格（如适用）后再付款。

保存确认邮件并完成考前检查

保存预约确认信息；在线考试再次运行系统测试并阅读签到规则；线下考试核对考试中心地址、到场时间与证件要求。若需要考试便利条件，应在预约前申请。

报名时的 6 个防错检查

贯穿原则

负责任 AI

公平、可靠与安全、隐私与安全、包容、透明、问责。先辨认风险，再匹配控制思路。

场景辨析

分析 vs 生成 vs 提取

分析是理解已有内容；生成是创建新内容；提取是把非结构化内容映射成字段或结构化输出。

做题时的 5 步法

  1. 圈出输入模态：文本、图像、音频、视频、文档/表单。
  2. 圈出目标动作：分类、检测、识别、摘要、生成、合成、提取、代理执行。
  3. 判断是否需要结构化输出：若需要字段/schema，优先想到信息提取或 Content Understanding。
  4. 判断实现层：概念选择、Portal 试验、模型 Endpoint、SDK 客户端、Agent。
  5. 检查责任 AI：是否涉及偏差、隐私、可解释性、安全性、无障碍或责任归属。

领域 1｜识别 AI 概念和能力（40–45%）

本领域不是让你推导复杂数学公式，而是要求你准确描述概念，并能从场景中识别合适的 AI 能力。

1.1 负责任 AI 六原则

原则你要能描述什么典型场景提示词常见误区
公平 Fairness系统应避免因数据、标签或设计造成对不同群体的不合理差异不同群体错误率差异；训练数据代表性；偏差测试公平不等于所有人得到完全相同结果，而是避免不当偏差
可靠性与安全 Reliability & Safety在预期与异常条件下保持稳定，识别失效模式并设置防护边界条件、鲁棒性、故障恢复、人工兜底准确率高不等于在所有场景都安全
隐私与安全 Privacy & Security保护个人/敏感数据，控制访问、传输、存储与滥用风险最小权限、敏感数据、加密、数据泄露隐私不仅是“隐藏姓名”，还涉及整个数据生命周期
包容 Inclusiveness让不同能力、语言、背景的用户都能有效使用无障碍、字幕、替代输入、多语言包容不等于公平；包容强调可访问和广泛参与
透明 Transparency让用户知道在与 AI 交互，说明能力、限制、数据使用和结果依据披露 AI、可解释性、限制说明、来源透明不等于公开模型全部源代码
问责 Accountability明确由人和组织对设计、部署与结果负责，建立治理与审查责任人、审批、审计、人工监督不能把责任“交给模型”
快速辨析：公平、包容、透明怎么区分？
  • 公平：不同群体是否受到不合理差别影响。
  • 包容：不同能力与背景的人是否都能使用、参与并受益。
  • 透明：用户是否理解这是 AI、它如何被使用、限制是什么。

1.2 AI 模型组件与配置

生成式 AI 模型如何工作

基础模型从大量数据中学习模式。对于文本生成，输入会被切分为 token，模型根据上下文预测后续 token；提示、系统指令、上下文和模型参数共同影响输出。推理（inference）是使用已训练模型生成结果，而训练/微调是改变模型权重或适应特定任务。

  • Prompt：传递任务、上下文、约束和输出格式。
  • System instruction：定义模型或代理的总体角色、规则和边界。
  • Context window：一次请求中模型可考虑的上下文范围。
  • Grounding：用可靠数据源约束回答，减少无依据输出。
  • Temperature：通常越低越稳定/保守，越高越多样；它不保证事实正确。
  • Max tokens：限制输出长度或生成预算。
如何按功能选择模型
需求优先考虑理由
文本问答、总结、代码或代理推理语言/生成模型擅长自然语言理解与生成
同时理解文字和图片多模态模型可在同一提示中处理视觉输入
生成新图片图像生成模型输出新的视觉内容，不只是识别已有图像
高稳定的分类/字段提取任务模型或信息提取工具输出更结构化、便于自动化
部署选项与配置：考试应掌握的判断
  • 模型目录（Model catalog）：发现并比较模型，按能力和任务选择。
  • Deployment：把所选模型部署为可调用资源；应用通常通过 Endpoint 和凭据访问。
  • Portal/Playground：适合探索模型、调整提示和参数、验证输出。
  • SDK/REST API：适合把能力集成进应用并自动化调用。
  • 吞吐、延迟、成本和区域：属于部署决策的重要约束；题目通常要求选择“更符合场景”的选项。

1.3 识别 AI 工作负载

工作负载核心动作示例不要混淆
生成式 AI基于提示创建文本、图像或其他新内容起草回答、生成图片、总结生成不是简单查找原文
代理 AI模型结合指令、上下文和工具，规划或执行任务调用工具查询数据、完成多步任务Agent 不只是聊天界面
文本分析/NLP从文本识别语义与结构关键短语、实体、情感、摘要情感分析不是语音情绪识别
语音语音转文本、文本转语音或语音交互字幕、听写、语音助手识别 speech-to-text；合成 text-to-speech
计算机视觉理解图像/视频中的对象、内容和关系分类、检测、OCR、视觉问答图像分类给整体标签；对象检测还定位对象
信息提取把非结构化内容转成预定义字段或结构发票字段、合同条款、音视频摘要字段OCR 只读文字；信息提取还要映射结构
文本分析四个高频概念
  • 关键短语提取：找出代表文本主题的重要短语。
  • 实体识别：识别人名、地点、组织、日期等实体。
  • 情感分析：判断文本表达的积极、消极或中性倾向。
  • 摘要：压缩原文并保留主要信息；生成式摘要可能改写表达。
视觉高频辨析
  • 图像分类：回答“整张图是什么”。
  • 对象检测：回答“图里有什么、在哪里”，通常包含边界框。
  • OCR：从图像中提取字符文本。
  • 多模态理解：结合图片与文字提示解释视觉内容。
  • 图像生成：根据提示创建全新视觉输出。

领域 2｜使用 Microsoft Foundry 实现 AI 解决方案（55–60%）

这是权重更高的领域。要理解“在 Foundry 中试验 → 部署 → 通过 SDK/API 调用 → 构建轻量应用或 Agent”的基本链路。

2.0 Foundry 核心对象

Project / Resource

组织 AI 开发资源、连接与项目工作。考试重点是理解它提供统一开发入口，而不是背所有管理字段。

Model catalog / Deployment

从模型目录选择适合任务的模型，并部署为应用可调用的模型端点。

Playground

交互式测试提示、参数和模型响应；验证后再把配置迁移到代码。

Endpoint / SDK / API

Endpoint 是调用入口；SDK/REST API 是应用集成方式；凭据与权限用于安全访问。

Agent

把模型行为、系统指令和工具封装起来，以完成带行动能力的任务。

Foundry Tools

提供语音、语言、Content Understanding 等任务能力，适合构建专门工作负载。

2.1 生成式 AI 应用和代理

有效提示的结构

系统提示（System）

规定长期角色、行为边界、优先级、语气、安全规则和输出约束。适合“始终遵守”的要求。

用户提示（User）

提供当前任务、输入、上下文和期望输出。适合本轮变化的业务需求。

推荐公式：角色/目标 + 输入上下文 + 明确任务 + 约束条件 + 输出格式 + 必要示例。提示越明确，不代表事实就一定正确；仍需 grounding、验证与责任 AI 控制。
部署模型并交互：最小流程
  1. 在 Foundry 中建立或进入项目。
  2. 从模型目录按任务选择模型。
  3. 部署模型，获得可调用的 deployment/endpoint。
  4. 在 Playground 中测试 system/user prompt 与参数。
  5. 查看代码示例，通过 SDK 或 REST API 构建轻量聊天客户端。
  6. 在应用中处理输入、调用端点、显示输出，并加入错误处理和安全控制。
创建和测试单代理解决方案
  • 定义代理的系统指令与目标。
  • 选择底层模型并配置可用工具。
  • 在 Portal/Playground 中验证代理是否按预期调用工具和回答。
  • 通过 Foundry Project API/SDK 从轻量客户端调用代理。
  • 测试正常路径、模糊输入、错误工具结果和不安全请求。

考试辨析：模型负责生成/推理；工具连接外部能力；代理把模型、指令和工具组织成可执行行为。

2.2 文本与语音 AI 解决方案

文本分析轻量应用

题目可能要求你在通用模型与 Azure Language/Foundry Tools 之间判断。通用模型适合开放式理解和生成；专门的文本分析能力适合稳定地提取情感、实体、关键短语等结果。

  1. 准备待分析文本。
  2. 选择通用模型提示或专用文本分析能力。
  3. 通过 Foundry 中的工具或已部署 Endpoint 试验。
  4. 用 Python SDK/API 发起请求。
  5. 解析响应并以结构化方式呈现。
语音识别、语音合成与多模态语音提示
能力输入输出典型场景
Speech-to-text音频/语音文本字幕、听写、转录
Text-to-speech文本合成语音语音播报、无障碍阅读
多模态模型响应语音提示语音（可能结合文字）模型理解后的响应自然语音交互

官方学习路径要求能使用 Azure Speech in Foundry Tools 构建轻量应用，并理解语音识别与合成的基本 API 方向。

2.3 计算机视觉与图像生成

用多模态模型解释视觉输入

将图片与文本问题一起提交给支持视觉的模型，用于描述、问答、分类或理解关系。高分答案应说明：输入包含图像与提示；选择支持视觉的多模态模型；在 Playground 验证；通过 SDK/API 调用。

使用生成模型创建视觉输出

图像生成的目标是创建新内容。提示应描述主体、构图、风格、光线和限制。与对象检测不同，输出不是标签或边界框，而是新的图像。

视觉轻量应用的最小架构
  1. 收集或上传图像/文本提示。
  2. 选择多模态理解模型或图像生成模型。
  3. 部署并在 Foundry Playground 测试。
  4. 客户端编码/引用图像并发送请求。
  5. 解析文本或图像响应，处理内容安全与错误。

2.4 使用 Content Understanding 实现信息提取

Azure Content Understanding in Foundry Tools 用生成式 AI 处理文档、图像、音频和视频，把非结构化内容转换成用户定义的结构化输出。核心不是只“看懂内容”，而是按 schema 提取、分类或生成字段。

对象含义考试抓手
Analyzer定义如何处理特定内容与生成结果可使用预构建或自定义分析器
Schema定义期望输出字段、类型和结构决定“要提取成什么样”
Extract提取内容中直接存在的值例如发票号、日期（文档分析器支持）
Classify从预定义类别中判断内容类型例如合同/发票/收据
Generate根据内容推理或总结生成字段例如摘要、风险说明
Grounding / Confidence追溯字段来源并评估置信度用于复核与人工审查
四种输入模态如何思考
  • 文档/表单：提取标题、表格、键值对和业务字段。
  • 图像：从照片或扫描件中获取文字与视觉字段。
  • 音频：先转录并可区分说话人，再生成摘要、情感或关键主题等结构化字段。
  • 视频：结合视觉与音频内容生成结构化洞察。
高频陷阱：OCR 负责把图像中的字符变成文本；Content Understanding 可以进一步把文本及其他模态内容映射到定义好的字段、分类和结构化结果。题目要“字段化输出”时，不要只选 OCR。

最后 48 小时只看这 10 组辨析

  1. 公平 vs 包容 vs 透明
  2. 可靠性/安全 vs 隐私/安全
  3. 训练 vs 推理
  4. system prompt vs user prompt
  5. 模型 vs deployment vs endpoint
  6. 模型 vs agent vs tool
  7. 语音识别 vs 语音合成
  8. 图像分类 vs 对象检测
  9. OCR vs 信息提取
  10. 分析已有内容 vs 生成新内容

考试作答建议

  • 先看题目最后一句，明确它问的是“概念、服务、实现步骤还是责任 AI 原则”。
  • 优先选择满足需求的最直接能力，不要因某个选项功能更多就选它。
  • 看到“字段、schema、结构化输出、文档/音频/视频提取”，重点考虑 Content Understanding。
  • 看到“调用外部能力、执行动作、多步任务”，重点考虑 Agent + Tools。
  • 看到“在哪里试提示和参数”，重点考虑 Playground；看到“集成应用”，重点考虑 SDK/API + Endpoint。
